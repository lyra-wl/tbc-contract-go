# tbc-contract-go 合约库说明

本仓库为 **TBC（TuringBitChain）链上合约与交易构造** 的 Go 语言实现，与 TypeScript 参考实现 **`tbc-contract`**（`lib/contract`、`lib/util` 等）逐函数对齐，便于在服务端、CLI 或业务集成中以原生 Go 调用。

> 手工与场景级验证见 **`docs/test-cases/`**：每篇 **开头为参数定义（环境变量）**，随后为 **单文件可 `go run` 的最小示例**（调用对应功能函数）；规范原文见并列仓库 **`../tbc-contract/docs/`**。

## 模块与依赖

- **Go module**：`github.com/sCrypt-Inc/tbc-contract-go`
- **最低 Go 版本**：`1.17`
- **核心依赖**：
  - `github.com/sCrypt-Inc/go-bt/v2`：交易、脚本、签名与 UTXO 模型（本地 `go.mod` 中通过 `replace` 指向 sibling `tbc-lib-go`，与团队内部 fork 对齐）
  - `github.com/libsv/go-bk`：密钥与密码学（`bec`、`crypto` 等）
  - `golang.org/x/crypto`：部分哈希/算法（如订单簿相关）

在仅克隆本仓库时，需同时准备被 `replace` 的 `go-bt` 路径，或按项目约定调整 `go.mod` 中的 `replace` 指令后再执行 `go build ./...`。

## 目录结构（库代码）

| 路径 | 说明 |
|------|------|
| `lib/contract/` | 合约与交易构造主体：FT、NFT、稳定币、订单簿、多签、HTLC、存钱罐、Pool NFT 2.0 等 |
| `lib/api/` | 面向 TBC 索引/API 的 HTTP 客户端：余额、UTXO、交易拉取、广播、FT/NFT/池子/冻结 UTXO 等 |
| `lib/util/` | 与 TS `lib/util` 对齐的工具：十进制与 `big.Int`、FT UTXO 解析、NFT 解锁脚本辅助等 |

## 包：`github.com/sCrypt-Inc/tbc-contract-go/lib/contract`

单包 `contract`，注释中标明与各 `*.ts` 的对应关系。主要能力如下。

### 同质化代币（FT）— `ft.go`

- **`FT`**：名称、符号、精度、总供应、`CodeScript` / `TapeScript`、合约 `txid` 等。
- **`NewFT(txidOrParams)`**：参数可为合约 `txid`（`string`）或新建参数 **`FtParams`**（`Name`、`Symbol`、`Amount`、`Decimal`）。
- **`(*FT).Initialize(*FtInfo)`**：用链上/索引拉取的元数据填充实例。
- **铸造 / 转账 / 合并等**：与 `tbc-contract` 中 FT 流程一致（具体方法名以源码为准，如 `MintFT`、`TransferFT`、多步合并等）。
- **诊断**：`ft_transfer_steps.go` 提供 **`FTTransferStepReport`** 等结构，用于与 JS 侧逐步对照签名摘要、线格式哈希与最终 `txid`。

### 稳定币（StableCoin）— `stablecoin.go`

- **`StableCoin`** 内嵌 **`*FT`**，复用 FT 层能力。
- **`NewStableCoin`**：构造方式与 **`NewFT`** 相同。
- 典型方法（与 TS `stableCoin.ts` 对齐）：**`CreateCoin`**（coin NFT + mint）、**`MintCoin`**、**`TransferCoin`**、**`FreezeCoinUTXO`** / **`UnfreezeCoinUTXO`**、**`MergeCoin`** 等。
- 链上 tape 中 `decimal` 等字段需满足 **`SCRIPT_VERIFY_MINIMALDATA`**；实现中对小整数 push 形式与 TS 栈行为对齐。

### NFT — `nft.go`

- **`NFT`**、**`NewNFT(contractID)`**、**`Initialize(*NFTInfo)`**。
- 集合与单枚元数据：**`CollectionData`**、**`NFTData`** 等，与 `nft.ts` 字段一致。
- 不含旧版非线性 **`poolNFT`**；NFT 2.0 流动性池见下文 **`poolnft2.go`**。

### 订单簿（OrderBook）— `orderbook.go`

- **`OrderBook`**、**`NewOrderBook()`**。
- 卖单/买单构造、撮合相关交易构建（脚本模板以常量形式内嵌，与 TS 侧字节布局一致）。
- 体量与边界条件复杂，建议结合 **`../tbc-contract/docs/orderBook.md`** 与 TS 参考实现理解输入输出。

### 多签 — `multisig.go`

- 地址与参数：**`GetMultiSigAddress`**、**`GetSignatureAndPublicKeyCount`**、**`VerifyMultiSigAddress`**。
- 与 JS **`MultiSig`** 一致的发送 TBC、转 FT 等交易构造（返回结构如 **`MultiSigTxRawSendTBC`**、**`MultiSigTxRawTransferFT`**）。

### HTLC — `htlc.go`

- **`DeployHTLC`**、**`Withdraw`**、**`Refund`** 等；交易使用与 FT 一致的 **`newFTTx()`**（版本等与 TBC 节点广播要求对齐，注释中说明与默认 `tbc-lib-js` 版本差异）。

### 存钱罐（定时锁 TBC）— `piggybank.go`

- **`GetPiggyBankCode`**、**`FreezeTBC`**、**`UnfreezeTBC`** 等与 `piggyBank.ts` 对齐；冻结手续费估算逻辑与 TS 一致（固定 80 sat/KB 规则，见源码注释）。

### Pool NFT 2.0 — `poolnft2.go`

- 仅实现 **Pool NFT 2.0（线性池）**，对应 `poolNFT2.0.ts`；旧版 pool NFT 故意未迁移。
- **`PoolNFT2`**、**`NewPoolNFT2(*PoolNFT2Config)`**、**`InitCreate`** 等；完整 swap/merge 等路径若仍在演进，以源码与 **`../tbc-contract/docs/poolNFT2.0.md`** 为准。

## 包：`github.com/sCrypt-Inc/tbc-contract-go/lib/api`

对默认 **主网** `https://api.turingbitchain.io/api/tbc/` 与 **测试网** `https://api.tbcdev.org/api/tbc/` 发起请求；**`network`** 参数通常传 **`"mainnet"`**、**`"testnet"`** 或自定义以 **`/`** 结尾的 base URL。部分 GET 在瞬时断连时会自动重试。

常用导出函数包括：

- **链通用**：`GetTBCBalance`、`FetchUTXO`、`FetchUTXOWithAPITxID`、`FetchUTXOs`、`GetUTXOs`、`FetchTXRaw`、`FetchTXRawHex`、`BroadcastTXRaw`、`BroadcastTXsRaw`、`IsTxOnChain`、`FetchBlockHeaders`
- **FT**：`GetFTBalance`、`FetchFtUTXOList`、`FetchFtUTXO`、`FetchFtUTXOs`、`FetchFtUTXOsForPool`、`GetFtUTXOsMultiSig`、`FetchFtInfo`、`FetchFtPrePreTxData`
- **NFT**：`FetchNFTUTXO`、`FetchNFTUTXOs`、`FetchNFTInfo`、`FetchNFTs`
- **稳定币索引**：`FetchStableCoinInfo`、`StableCoinIndexerIDFromMintContractTx`（余额查询在实现中与 FT 索引路径复用内部逻辑）
- **池子**：`FetchPoolNFTInfo`、`FetchPoolNFTUTXO`、`FetchFtLpBalance`、`FetchFtLpUTXO`
- **冻结 UTXO（存钱罐等）**：`FetchFrozenTBCBalance`、`FetchFrozenUTXOList`、`FetchUnfrozenUTXOList`、`FrozenToBTUTXO`
- **自定义脚本 UTXO**：`FetchUMTXO`、`FetchUMTXOs`、`GetUMTXOs`

各响应结构体（如 **`FtInfo`**、**`NFTUTXO`**、**`PoolNFTInfo`**）定义于对应 `api_*.go` 文件中。

## 包：`github.com/sCrypt-Inc/tbc-contract-go/lib/util`

与 TS `lib/util` 行为对齐的辅助方法，例如：

- **`ParseDecimalToBigInt(amount, decimal)`**：小数字符串按精度转为 `*big.Int`
- **`IsValidSHA256Hash`**、**`IsValidHexString`**
- **`BuildUTXO`**、**`FtUnspentOutput`**：从交易输出构造带 FT 余额信息的 UTXO
- **`nftunlock.go`**：**`NftEncodeMinimalPushData`** 等与链上 **`SCRIPT_VERIFY_MINIMALDATA`** 一致的 push 编码，供 NFT / coin NFT 解锁路径使用

## 环境变量（可选，多用于调试或与节点行为对齐）

在 **`lib/contract`** 的 FT 等路径中会读取（若未设置则走默认逻辑），包括但不限于：

- **`FT_FEE_SAT_PER_KB`**、**`NFT_FEE_SAT_PER_KB`**：估算手续费时的 sat/KB
- **`FT_SIGNED_UNLOCK_BYTES`**、**`FT_RELAY_FEE_SIGNED_ESTIMATE`**、**`FT_RELAY_SIGNED_UNLOCK_BYTES`**：与带签名解锁脚本大小、中继估算相关的调试开关

具体语义以 **`ft.go`** 等文件内注释为准。

## 构建说明

本仓库仅保留 **`lib/`** 下供引用的 Go 源码（及合约模板等被 `//go:embed` 的资源），**不包含** 包内 `*_test.go` 与对照脚本。行为验证请在业务仓库编写测试，并对照 **`tbc-contract`** 与 **`docs/test-cases/`**。

## 在项目中引用

```go
import (
    "github.com/sCrypt-Inc/tbc-contract-go/lib/api"
    "github.com/sCrypt-Inc/tbc-contract-go/lib/contract"
    "github.com/sCrypt-Inc/tbc-contract-go/lib/util"
)
```

典型流程：**`api.FetchUTXO` / `FetchFtUTXO`** 取得输入 → **`contract`** 中对应类型构造交易 → **`bec.PrivateKey` + `unlocker`** 填签名 → **`api.BroadcastTXRaw`** 广播。私钥与 `txraw` 的安全存储与传输不在本库范围内，由调用方负责。
